# AgentChat · 总体架构设计

> 一个参考 claude.ai 交互形态、后端对接 Claude Agent SDK (Python) 的自定义聊天应用。
> 定位：**claude.ai 的外观 + Claude Code 的引擎**。单页对话交互，覆盖文件操作、Bash、
> 斜杠命令、Subagent、MCP、Hook、细粒度权限等 Agent 能力。

## 1. 系统分层

```
┌─────────────────────────────────────────────────────┐
│  前端 SPA (React)                                     │
│  会话列表 / 消息流 / 命令面板 / 审批卡片 / 设置中心     │
└──────────────┬──────────────────────────────────────┘
               │ HTTPS (REST 上行) + SSE (下行事件流)
┌──────────────▼──────────────────────────────────────┐
│  API 网关层 (FastAPI)                                 │
│  JWT 鉴权 · 路由 · 限流 · 用量校验 · 请求审计          │
├─────────────────────────────────────────────────────┤
│  应用服务层                                            │
│  ThreadService / RunService / CommandService          │
│  HookService / PermissionService / ApprovalService    │
├─────────────────────────────────────────────────────┤
│  Agent 执行层 (Agent Runner)                          │
│  封装 claude-agent-sdk：query()/resume/fork           │
│  运行前组装: options + hooks + 权限 + 物化命令文件      │
├──────────┬───────────────────┬──────────────────────┤
│ 元数据存储 │  会话工作空间       │  Session Transcript   │
│ SQLite    │ /workspace/{uid}/  │ SDK 管理的 JSONL      │
│ (→MySQL)  │   {thread_id}/     │ (与 cwd 绑定)         │
└──────────┴───────────────────┴──────────────────────┘
```

三个存储各司其职，边界必须清晰：

| 存储 | 内容 | 谁是真相源 |
|---|---|---|
| 元数据库 (SQLite→MySQL) | 用户、会话索引、消息副本、权限/Hook/命令配置、审计、用量 | 应用自身 |
| 会话工作空间 (文件系统) | Agent 可读写的隔离目录、上传文件、物化的 `.claude/skills/` | 应用生成，Agent 消费 |
| Session Transcript (JSONL) | Agent SDK 自动持久化的对话上下文，resume/fork 的依据 | Agent SDK |

**关键原则**：数据库里的 `messages` 表是给"UI 渲染 + 搜索 + 审计"用的副本；
Agent 恢复上下文永远走 SDK 的 `resume`（读 transcript），二者不要互相当作对方的来源。

## 2. 核心运行时流程

### 2.1 一次消息的完整生命周期

```
用户发送 → 校验(鉴权/额度/并发锁) → 落库 user message + 创建 run
  → Agent Runner 组装 options:
      cwd = thread.workspace_path
      resume = branch.agent_session_id (首轮为 None)
      model / thinking / allowed_tools ← 本轮请求参数
      hooks ← HookService.build(user, project)
      can_use_tool ← ApprovalService 的异步审批回调
      (运行前物化 slash_commands / subagents / mcp 配置到 cwd)
  → async for msg in query(...):
      system/init  → 捕获 session_id 写回 branch
      assistant    → 增量写入 run 事件流(SSE) + 缓冲落库
      tool_use     → 推送"工具调用卡片"事件
      tool_result  → 推送结果 + 审计日志
      result       → 结算 usage/cost, run=completed
  → 异常/取消 → run=error/stopped, 保留已生成部分
```

### 2.2 并发与实例模型

- 同一 `branch_id` 同时只允许一个 run（应用内 asyncio 锁；多实例时换 Redis 锁）。
- **第一期单实例部署**（SQLite 决定了这一点），stop/审批的 Future 都放进程内存即可。
- 迁 MySQL + 多实例时的改造点（提前留接口）：
  - 会话粘性路由（按 thread_id 一致性哈希）或 SessionStore 适配器把 transcript 落共享存储；
  - stop / 审批决议改走 Redis pub/sub 通知持有任务的实例；
  - SSE 事件生产改为经由消息总线扇出（任一实例可服务 stream 订阅）。

## 3. 前后端通信协议（SSE）

- REST：所有 CRUD + 所有"上行动作"（发送消息、停止、审批决定）。
- SSE：仅承载"下行事件"。核心是 **run 级事件流**：

```
POST /threads/{id}/messages  → {run_id}          (上行, 立即返回)
GET  /runs/{run_id}/stream   → text/event-stream (下行, EventSource 订阅)
POST /runs/{id}/stop         → 204               (上行)
POST /approvals/{id}/decide  → 204               (上行, 服务端 resolve 审批 Future,
                                                   run 流里随后推进后续事件)
```

选择 SSE 的收益：EventSource 原生自动重连 + `Last-Event-ID` 断点续传；
纯 HTTP，网关/代理/鉴权中间件全部复用，无需维护长连接会话状态。
代价：下行单向——但本应用的上行本来就都是离散动作，POST 足够。

统一的 SSE 事件格式（`id:` 为递增序号，供断线续传）：

```
id: 42
event: text_delta
data: {"thread_id":"…","run_id":"…","data":{"text":"…"}}
```

| type | 说明 |
|---|---|
| `text_delta` | assistant 文本增量 |
| `thinking_delta` | 思考内容增量（前端折叠展示） |
| `tool_use` | 工具调用开始 {tool_name, tool_input, tool_use_id} |
| `tool_result` | 工具结果 {tool_use_id, output, is_error} |
| `approval_request` | 需要用户审批 {approval_id, tool_name, tool_input} |
| `run_status` | running/completed/stopped/error + usage 结算; 流以此事件收尾后服务端关闭 |
| `title_update` | 自动生成的会话标题（随 run 流尾部下发, 或前端 run 结束后 REST 刷新） |

**断线续传**：服务端为每个进行中的 run 维护事件 ring buffer（带递增 id）；
EventSource 自动重连时浏览器带 `Last-Event-ID` 头，服务端从该序号回放。
run 结束后 buffer 即弃，历史消息一律走 REST 补齐。
**保活**：每 15s 发一行 SSE 注释 `: ping`，防止代理超时断连。

## 4. 部署形态

第一期（轻量）：

```
uvicorn app:app (单进程)        # FastAPI + SSE + Agent Runner 同进程
SQLite (WAL 模式)               # 数据文件随应用目录
/workspace/                     # 本地磁盘
Agent SDK 运行时               # pip 包已自动捆绑 Claude Code CLI，无需单独安装;
                               # 是否仍需 Node 取决于所装版本的捆绑形态，部署前确认
```

上线期（MySQL 后）：

```
nginx → N × uvicorn 实例（粘性路由 by thread_id）
  SSE 必配: proxy_buffering off + 响应头 X-Accel-Buffering: no
  proxy_read_timeout 调大（长流）；HTTP/2 下浏览器无 6 连接上限问题
MySQL 8.x + Redis(锁/PubSub) + 对象存储(上传文件/产物)
Agent 执行迁入独立容器池（每会话/每用户一容器，见 04 文档安全章节）
认证计费: 对外服务必须走 API key（Anthropic 政策不允许第三方产品
  向自己的终端用户提供 claude.ai 订阅登录）
```
